## Executive Summary

Nocturne is a well-architected event aggregation platform with a React frontend, FastAPI backend, and multi-source scraper (Eventbrite, Meetup, Luma). It currently supports 40+ US cities with automated scraping and email digests. 

**Key Finding:** The project is technically sound but is **leaving significant value on the table** by limiting itself to only 3 event sources when 10+ high-quality sources are available. The "underground" branding is compelling but underutilized in growth strategy.

**Immediate Opportunity:** Adding Dice.fm, Resident Advisor, and Songkick would increase event coverage by an estimated 300-400% for music/events, transforming this from a generalist scraper to the definitive underground event authority.

## New Source/Tool Research

### 1. DICE.FM - HIGH PRIORITY ⭐⭐⭐
**What it is:** The leading mobile ticketing platform for underground electronic music, live music, and club events in major cities.

**API Availability:** 
- GraphQL API at `https://partners-endpoint.dice.fm/graphql`
- Public JSON API available (no auth required for event discovery)
- Well-documented schema with Event, Venue, Artist, Ticket types

**Data Available:**
- Event name, description, dates, times
- Venue details (name, address, coordinates)
- Artist lineups and genres
- Ticket URLs and pricing tiers
- Event images/banners
- Genre classifications

**Integration Complexity:** LOW-MEDIUM
```python
# Dice.fm GraphQL query example
dice_query = """
{
  viewer {
    events(first: 50) {
      edges {
        node {
          id
          name
          startDatetime
          endDatetime
          description
          url
          venues {
            name
            address { street city state zip }
          }
          artists { name }
          images { url }
        }
      }
    }
  }
}
"""
```

**Value Add:** Dice.fm specializes in underground electronic, techno, house - exactly matching Nocturne's aesthetic. Would add 50-100 events per major city per week.

### 3. SONGKICK - MEDIUM-HIGH PRIORITY ⭐⭐
**What it is:** One of the largest live music databases (6M+ concerts), owned by Warner Music Group.

**API Availability:**
- Official API at `https://api.songkick.com/api/3.0/`
- Requires API key (free tier available)
- Well-documented REST API

**Data Available:**
- Artist info, event listings
- Venue details
- Metro area search (by city)
- Concert dates, times
- Ticket links

**Integration Complexity:** LOW (official API)

```python
# Songkick API endpoint pattern
metro_id = "7644"  # NYC
url = f"https://api.songkick.com/api/3.0/metro_areas/{metro_id}/calendar.json?apikey={API_KEY}"
```

**Value Add:** Strong for indie/alternative/rock shows that Eventbrite misses. Official API means reliable access.

### 4. PARTIFUL - MEDIUM PRIORITY ⭐⭐
**What it is:** The "cool" event platform for house parties, underground raves, and exclusive events. Very Gen Z.

**API Availability:**
- No official API
- Reverse-engineered: `github.com/cerebralvalley/partiful-api`
- Uses Firebase under the hood
- Requires auth token from browser (session-based)

**Data Available:**
- Event details, hosts, attendees
- Guest lists (with privacy implications)
- Mutual connections

**Integration Complexity:** MEDIUM-HIGH
- Auth token rotation required
- Unofficial API could break
- Privacy considerations for guest lists

**Value Add:** Partiful is the most "underground" of all sources - house parties, pop-ups, raves. However, privacy concerns and unofficial API make this risky.

**Recommendation:** Monitor but don't implement yet. Wait for official API or partnership.

## Structural Improvements

### 1. MICROSERVICES ARCHITECTURE
**Current:** Monolithic FastAPI + scrapers  
**Future:** Separate services:
- `api-service`: FastAPI backend
- `scraper-service`: Celery workers for scraping
- `notification-service`: Email/push notifications
- `ml-service`: AI enrichment (separate for scaling)

**Benefit:** Can scale scraper workers independently, ML service can use GPU instances.

### 2. MESSAGE QUEUE FOR SCRAPING
**Current:** Direct scraping calls  
**Better:** Redis + Celery task queue
- Retry failed scrapes automatically
- Rate limiting per source
- Parallel scraping across cities

### 3. EVENT SOURCING / DATA LAKE
**Current:** Events stored once  
**Better:** Track event changes over time:
- When was event added?
- When did price change?
- When did event sell out?

**Storage:** Add ClickHouse or TimescaleDB for time-series analytics.

### 4. GRAPHQL API LAYER
**Current:** REST API  
**Better:** GraphQL for frontend:
- Fetch only needed fields
- Reduce over-fetching
- Easier mobile app development

### 5. CDN FOR IMAGES
**Current:** Likely hotlinking to source images  
**Better:** Download and serve through CDN (Cloudflare Images, AWS CloudFront)
- Faster load times
- Images don't break when sources change
- Consistent sizing/formatting

## Warnings: Time Wasters & Bad Directions

### ⚠️ DO NOT DO THESE:

1. **Don't build your own ticketing system**
   - Too complex, regulatory issues, fraud risk
   - Affiliate to Dice, RA, Eventbrite instead

3. **Don't expand to 100 cities yet**
   - Focus on 5-10 cities with depth

4. **Don't build a social network**
   - Not the core value prop
   - Stick to event aggregation + light social features

5. **Don't chase paid partnerships with venues yet**
   - Get user base first
   - Venues will come to you once you have traffic

6. **Don't over-engineer the scraper**
   - The current Playwright/pydoll setup is fine
   - Don't rebuild in Rust/Go unless scale demands it
